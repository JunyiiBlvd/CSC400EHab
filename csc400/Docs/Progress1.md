1. Executive Summary
   E-Habitat is a self-proposed educational simulation platform developed as part of CSC400. The system enables computer science students and instructors to compare centralized and edge-based distributed monitoring architectures side-by-side within a virtual server room environment, with real-time anomaly injection and detection latency metrics. The project spans 16 weeks across four phases.
   This progress report covers Phases 1 and 2 (Weeks 1–8), documents implementation status relative to SRS v1.2 (corrected to reflect the full 16-week timeline), identifies active gaps, and outlines the plan for Phases 3 and 4 (Weeks 9–16). With eight weeks remaining, all Must Have user stories are achievable within the remaining schedule.

4.1 What Is Working
The core educational value of E-Habitat is fully functional. The physics simulation produces realistic environmental dynamics validated against real-world dataset distributions. The Isolation Forest model reliably detects injected anomalies with no false positives observed since threshold tuning in Week 4. The architecture comparison panel is live on the dashboard. The UI stack is aligned with SRS v1.2.
Importantly, the work that is complete represents the novel and technically challenging parts of the project: the physics engine, ML pipeline, and distributed architecture comparison. The gaps are infrastructure features (auth, DB, WebSockets) that are well-understood engineering tasks with clear bounded scope.
4.2 What Is Behind
Authentication and SQLite persistence are the most significant gaps. Both are Must Have user stories in SRS v1.2 (US001, US002) and neither has been started. These were assigned in Phase 1 and 2 responsibility matrices to Logan (auth) and Gavin (database) but were deprioritized in favor of stabilizing the simulation and ML pipeline. With two full phases remaining, both items are achievable within schedule.
The REST-to-WebSocket migration is the highest-priority technical gap relative to the SRS architectural requirement. The existing endpoint structure makes the migration well-understood, and FastAPI natively supports WebSockets.
The single-node limitation in api.py is a straightforward backend expansion. The VirtualNode abstraction already supports independent operation; multiplexing three instances is primarily a backend concurrency task.
4.3 Intentional Divergence
The training data approach for the Isolation Forest model is noted as Diverged rather than Behind. The SRS originally specified merging real CSV rows into the training corpus. During Week 4, the team made the deliberate decision to use purely synthetic telemetry generated by the real-data calibrated physics engine, and the large CSV files were removed from the repository to reduce size.
This is a defensible implementation decision. The real datasets were used to set the physics parameters that govern the synthetic data generation, making the training data statistically grounded in real-world distributions. The SRS v1.2 has been updated to reflect this as the documented approach. It is not a scope failure. 5. Phases 3 & 4 Plan (Weeks 9–16)

8.1 What Is Working Well
• Starting with the novel, technically risky components first (physics engine, ML pipeline) was the right call. These are now stable, and Phases 3 and 4 can focus on well-understood engineering tasks.
• The SRS revision process (v1.0 → v1.1 → v1.2) has been valuable. Each revision surfaced scope clarifications early rather than late. Building in explicit SRS checkpoints after each phase is recommended practice.
• The synthetic calibrated data approach for Isolation Forest training is sound and simplified the ML pipeline without sacrificing detection quality. Documenting this as an intentional decision (rather than a gap) was the right framing.
8.2 What Should Change
• Infrastructure features (auth, DB, WebSockets) should be scaffolded in parallel with core feature development, not deferred. The decision to defer auth allowed faster progress on the simulation core but created Phase 3 debt. On the next project, a minimal auth scaffold should be built in Week 1.
• The WebSocket-first architecture specified in the SRS should have been implemented in Week 2 as written. Retrofitting from REST polling in Phase 3 requires coordinated frontend and backend changes simultaneously. Building it correctly from the start would have been less costly overall.
• Timeline errors should be caught and corrected at the SRS level before they cascade. The 12-week vs. 16-week discrepancy between SRS v1.1 and the actual semester was corrected in SRS v1.2, but earlier alignment with the course calendar would have prevented the confusion. 9. Conclusion
E-Habitat is on track to deliver a complete, educational, and demonstrable distributed systems simulation platform by the Week 16 final demo. The core simulation physics engine, Isolation Forest anomaly detection, and centralized vs. edge architecture comparison is functional and stable after eight weeks of development.
The gaps identified in this report are real but manageable. Authentication, SQLite persistence, and WebSocket streaming are Must Have requirements that have not been started, but they are well-scoped, well-understood engineering tasks with clear ownership and eight weeks of remaining schedule. The Phase 3 and 4 plans provide a week-by-week remediation path with explicit scope contingencies.
The corrected 16-week timeline, reflected in SRS v1.2, also provides Phase 4 as a dedicated testing, polish, and UAT phase that did not exist in SRS v1.1. This is a meaningful improvement: it means the final two weeks before submission are reserved for quality rather than feature delivery, which substantially reduces the risk of a crash-and-burn final sprint.
The project's educational mission is being realized. The simulation makes distributed systems tradeoffs visible and interactive. The remaining work will make it complete.
