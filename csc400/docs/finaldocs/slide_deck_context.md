## SLIDE 5 — Architecture Diagram

**Title:** Dual-Path Architecture

**Key Points:**
- Both paths use the same trained Isolation Forest model
- **Edge path:** VirtualNode.step() → Feature extraction → ML predict → WebSocket push (full frame)
- **Central path:** Raw telemetry only → CentralServer → Feature extraction → ML predict → REST poll
- Latency delta = time between edge detection and central detection for the same anomaly
- Bandwidth difference = central sends ~48% of the bytes edge sends

## SLIDE 6 — Physics Simulation Engine

**Title:** PsyEngine — The Physics Behind the Nodes

**Key Points:**
- Custom physics engine: **ThermalModel** (Newton's Law of Cooling), **AirflowModel** (proportional HVAC control), **HumidityModel** (mean-reversion + temperature coupling)
- CPU load generated by an AR(1) stochastic process — realistic volatility
- All three physics models run in sequence every 1 second per node
- Parameters calibrated against real-world sensor data (MIT CSAIL Intel Lab + Kaggle HVAC + cold source server room control dataset)

**Key Equation (thermal, show simplified):**
```
ΔT = (P_heat − P_cool) / thermal_mass × dt
where P_cool = cooling_coefficient × airflow_ratio × (T − T_ambient)
```

**Context for designer:** This slide shows the simulation is grounded in physics, not random numbers. Keep the equation clean and small — it's supporting evidence, not the main point. The main point is: "the simulation behaves like a real server room."

**Visual:** A simple block diagram: [CPU Load AR(1)] → [Thermal Model] ↔ [Airflow Model] ↔ [Humidity Model] → [Output: temp, airflow, humidity, cpu_load]. Label arrows with coupling relationships.

## SLIDE 7 — ML Pipeline

**Title:** Anomaly Detection — Isolation Forest

**Key Points:**
- **Model:** Isolation Forest (unsupervised) — learns normal behavior, flags deviations
- **Why Isolation Forest:** Fast (real-time edge inference), unsupervised (no labeled anomalies needed), interpretable anomaly score
- **Feature extraction:** 12 statistical features from a 10-step sliding window
  - Mean, variance, rate-of-change for each of: temperature, airflow, humidity, CPU load
- **Detection threshold:** `score < 0.15`
  - Normal baseline floor: ~0.23 (zero false positives in testing)
  - ~0.08 gap between normal floor and threshold = safety margin
- **Trained on a 22,239-row hybrid dataset:**
  - 45% synthetic (PsyEngine baseline), 22.5% MIT CSAIL Intel Lab, 16.9% Kaggle HVAC, **15.7% cold source server room control data**
  - Cold source dataset was found by Gavin — real server room telemetry; no humidity column, imputed from MIT CSAIL statistics

**Context for designer:** Show the pipeline as a flow: Raw telemetry → 10-step sliding window → 12 features → Scaler → Isolation Forest → anomaly_score + is_anomaly. Label the 12 features simply as "mean, variance, rate-of-change × 4 sensors." Optionally add a small pie chart of the four training data sources.

**Visual:** Horizontal pipeline flow diagram. Below it: a small score axis showing "Normal: +0.23 → …" and "Threshold: 0.15" and "Anomaly: < 0.15" with color coding (green/red).

## SLIDE 8 — Real-World Data Validation

**Title:** Does the Model Generalize to Real Anomalies?

**Key Points (answer upfront: YES):**
- Model was trained entirely on normal operation data — it has never seen anomalies
- Tested against 1,732 MIT CSAIL Intel Lab anomaly windows + 134,516 in-distribution normal windows

**Full metrics (retrained model, April 2026):**

| Metric | Result |
|---|---|
| Recall | **100.00%** (1,732 / 1,732 detected) |
| Precision | **97.58%** (only 43 normal windows misclassified) |
| F1 Score | **98.77%** |
| Specificity | **99.97%** |
| False Positive Rate | **0.032%** |
| AUC-ROC | **1.0000** |

- Score gap: anomaly windows avg +0.005, normal windows avg +0.269 — at the 0.15 threshold the two populations barely overlap
- Original model achieved 95.96% recall; retraining after cold source humidity calibration pushed it to 100%

**Context for designer:** This is the credibility slide. Present the metrics table cleanly. The 100% recall and 1.0 AUC are the headline numbers. F1 98.77% shows precision is nearly as strong. Use the score distribution gap to visually illustrate separation.

**Visual:** Two-column layout: left = metrics table; right = score distribution visualization showing two peaks (anomaly cluster near +0.005, normal cluster near +0.27) with the 0.15 threshold line drawn between them.

## SLIDE 12 — Measured Results: Latency

**Title:** Results — Detection Latency

**Key Points:**
- Both paths detect anomalies after the 10-step sliding window fills (~10 seconds after injection)
- Edge path detects **~38–42 ms earlier** than the centralized path
- Latency range: 2,500–4,070 ms (varies with window fill state at injection)
- The delta varies 10–1,000 ms depending on how full the sliding window is at injection time — this variance is correct and academically meaningful

**Interpretation:**
- The difference is small in absolute terms but consistent in direction
- In low-latency industrial applications (power grid, medical), even tens of milliseconds matter
- For educational simulation: the architectural reason *why* it differs is the teaching point

**Context for designer:** A horizontal bar comparison: "Edge: 2,500–4,070 ms" vs. "Central: 2,550–4,108 ms." Annotate the delta (~38–42 ms) with an arrow. Include a note: "Variance explained by window fill state."

## SLIDE 14 — Known Limitations

**Title:** Honest Limitations

**Key Points:**
1. **HVAC failure required workaround:** Instant fan cutoff wasn't ML-detectable — required a 15-step ramp design. This means "HVAC failure" in the demo is a slow degradation, not a sudden failure.
2. **bytes_edge vs. bytes_central measure different things:** Not directly comparable per-frame — one is a snapshot, one is a cumulative. The ratio is correct; the raw numbers are not apples-to-apples.
3. **Model does not diagnose root cause:** It flags "something is wrong" — not "HVAC failed" or "coolant is leaking." Root-cause analysis would require labeled training data.
4. **Latency delta is small:** 38–42 ms may not be meaningful for all applications. For educational demonstration, the direction of the result is the key finding.
5. **Cold source imputation created artificial high-frequency noise:** The cold source dataset had no real humidity column, so it was imputed with i.i.d. draws from N(38.63, 7.21) — the right marginal distribution but no temporal correlation. Real sensors drift by fractions of a unit per step; i.i.d. sampling from σ=7.21 makes consecutive readings jump ~7.21 RH per step. The expected sliding-window variance of those i.i.d. draws is σ²≈52 — which matches the observed 49.0 exactly. The model correctly flags this as anomalous noise. The fix is a correlated time-series (AR(1) or random walk), not i.i.d. sampling. Excluded from FP metrics for this reason.
6. **Local deployment only:** No external server, no authentication, no network hardening. Designed for local demo, not production use.

**Context for designer:** A "limitations" slide that reads as thoughtful, not defensive. Use a neutral icon (caution, not error). The framing should be "we know what we don't know."

## SLIDE 15 — Lessons Learned

**Title:** What We Learned

**Key Points (team perspective):**
- **Physics calibration is non-trivial:** The cold source dataset (15.7% of training data) had no humidity column — it was filled with arbitrary synthetic parameters. The professor flagged this. Recalibrating to MIT CSAIL statistics (μ=38.63, σ=7.21) caused recall to collapse from 95.96% to 15.53% in one failed retrain attempt, then a second attempt dropped it further. Root cause: the validation script was using the model's internal contamination threshold, not the deployed 0.15 threshold. Fixing the gate and retraining pushed recall to 100%.
- **Unsupervised ML is sensitive to training distribution:** The model learns what "normal" looks like. Any change to normal-operation parameters requires retraining. Physics and model are tightly coupled.
- **Threshold is not the same as model output:** The model's internal contamination offset (−0.6134) and our deployed detection threshold (0.15) are different. This caused a false validation failure — a subtle bug with a non-obvious cause.
- **Ramped anomalies are more detectable than instantaneous ones:** The sliding window feature extractor needs multiple deviant readings to build a strong feature signal. A one-step jump may not move the mean/variance enough to exceed threshold.
- **The comparison panel is the academic core:** Building CentralServer was the riskiest deliverable. We built it first in Phase 3 for exactly this reason.

**Context for designer:** Conversational bullet points. No diagrams needed. This should feel like a genuine reflection, not a requirements checklist.
