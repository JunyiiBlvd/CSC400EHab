## 1. Executive Summary

### 1.1 Project Overview

E-Habitat is an interactive educational simulation platform designed to help learners understand distributed systems concepts by comparing centralized and edge-based monitoring architectures in a realistic server room environment. The system simulates thermal, airflow, and humidity dynamics across multiple virtual nodes and visualizes how different monitoring strategies detect anomalies under identical conditions. E-Habitat enables hands-on exploration of architectural tradeoffs without requiring physical infrastructure.

### 1.2 Problem Statement

Students studying distributed systems often encounter abstract concepts such as edge computing, centralized monitoring, latency, and bandwidth tradeoffs without access to realistic experimentation environments. Existing tools are either production-grade monitoring systems that are too complex for educational use or purely theoretical materials that lack experiential learning. This gap limits students' ability to develop intuition about real-world distributed system behavior. E-Habitat bridges this gap by providing a controlled, visual, and interactive simulation environment.

### 1.3 Solution Overview

E-Habitat addresses this gap by providing a controlled, visual, and interactive simulation where centralized and edge monitoring architectures operate simultaneously on the same environmental data. By allowing users to inject anomalies and observe detection latency, bandwidth usage, and system behavior in real time, the platform makes distributed systems principles concrete and observable.

### 1.4 Client Context

**Client Organization:** Self-Proposed
This project is self-proposed and intended for use by computer science students, instructors, and self-learners in academic settings. The primary success criteria are educational clarity, ease of setup, and the ability to demonstrate architectural tradeoffs during live demos or guided lab exercises. Requirements and scope were validated against typical distributed systems course objectives and instructor-led demonstration needs.

## 4. Features & Requirements

### 4.1 Core Features

1. **Simulation Engine** — Simulates thermal, airflow, and humidity dynamics across three virtual nodes at one-second intervals.
2. **Dual Monitoring Architectures** — Centralized raw-telemetry streaming and edge-based local anomaly detection running concurrently.
3. **Anomaly Scenarios** — Two predefined anomaly scenarios (thermal spike, HVAC failure) with manual injection controls.
4. **Real-Time Dashboard** — Displays telemetry, alerts, and architecture comparison metrics.
5. **Educational Comparison Metrics** — Detection latency and network message volume for each architecture.

### 4.2 Technical Requirements

#### Data Management

- **Read:** Users can see node behavior react to changing environmental parameters
- **Update:** Users can modify parameters in active simulations

#### Database & Storage

- **Primary Database:** SQLite
- **Key Data Entities:** Telemetry records, anomaly events, injection log
- **Telemetry:** Per-node sensor readings with timestamps
- **Anomaly Events:** Detection timestamps for both edge and centralized architectures, bytes transmitted per architecture

#### Simulation Engine

- The system shall simulate at least three virtual nodes concurrently.
- The system shall stream telemetry using WebSockets.
- The system shall compute anomaly scores using a pre-trained Isolation Forest model.
- The system shall display alerts when anomaly thresholds are exceeded.
- Both edge and centralized architectures shall run concurrently on identical sensor data.

### 4.3 Non-Functional Requirements

#### Performance

- Dashboard updates shall occur with less than 200ms latency
- Support for at least three nodes without dropped messages

#### Usability

- Interface shall support live demos without prior configuration
- Core actions shall be accessible within two clicks
- UI elements should be intentionally placed with simplicity in mind
- Optimization for dark mode theme
- All core information should be available at-a-glance

#### Reliability

- System shall run continuously for at least one hour without failure
- Error messages should be used when appropriate
- ML engine should avoid false positives during normal operation

### 5.4 System Architecture

**Architecture Pattern:** WebSocket streaming (edge) + REST polling (centralized simulation)

**Component Diagram:**

```
[FastAPI backend :8000]
    ├── nodes dict — 3 VirtualNode instances (module-level, persistent)
    ├── central_server — CentralServer instance (module-level)
    ├── ws://localhost:8000/ws/simulation   push 1 frame/sec (edge telemetry)
    ├── POST /simulation/inject             trigger anomaly scenarios
    └── GET /central/status                 centralized inference results (polled)

[VirtualNode.step() — every 1 second per node]
    1. AR(1) cpu_load (or spike override)
    2. AirflowModel.step(temperature)
    3. ThermalModel.step(cpu_load, airflow_ratio)
    4. HumidityModel.step(temperature)
    5. SlidingWindowFeatureExtractor
    6. ModelLoader.predict() → anomaly_score, is_anomaly (EDGE inference)
    7. Anomaly persistence window (20 steps)
    8. Return telemetry + seq_id + timestamp

[CentralServer — same process, module-level]
    ├── Receives raw telemetry (no anomaly score)
    ├── Maintains separate sliding windows per node
    ├── Runs shared ModelLoader (same model as edge)
    └── Exposes GET /central/status polled at 1Hz by frontend

[Next.js frontend :3000]
    ├── WebSocket client — demux by node_id
    ├── Three node panels with live charts
    ├── AlertsFeed — ML-only alerts, transition-based
    ├── Controls panel — inject buttons, sliders
    └── Comparison panel — latency delta and bandwidth ratio
```

**Request/Response Flow (Edge):**

1. VirtualNode.step() runs every 1 second
2. ML inference happens inside the node before telemetry leaves
3. WebSocket pushes frame to frontend with is_anomaly already set
4. Frontend fires alert on false → true transition

**Request/Response Flow (Centralized):**

1. VirtualNode.step() runs every 1 second
2. Raw telemetry (no inference) sent to CentralServer in same process
3. CentralServer runs inference on received telemetry
4. Frontend polls GET /central/status every 1 second
5. Latency delta = centralized_detection_time − edge_detection_time

### 6.2 Anomaly Detection Pipeline

**Training (Offline):**

1. Generate baseline telemetry using physics engine with real-data parameters
2. Extract sliding-window features (10-second windows)
3. Train Isolation Forest on normal operation only
4. Save model as `.pkl` artifact for deployment

**Inference (Real-Time During Simulation):**

1. Nodes collect telemetry in sliding windows
2. Extract features (mean, variance, rate-of-change per signal)
3. Run Isolation Forest prediction
4. If anomaly score exceeds threshold → trigger alert
5. Edge path: inference runs inside VirtualNode before telemetry is transmitted
6. Centralized path: raw telemetry transmitted to CentralServer, inference runs there

**Why Isolation Forest:**

- Unsupervised: doesn't require labeled anomalies
- Fast: suitable for real-time edge inference
- Interpretable: anomaly scores are understandable
- Proven: widely used in production anomaly detection

**Limitations Acknowledged:**

- Detects deviations from training distribution — does not predict future events
- Threshold calibrated against live simulation (distribution shift from training noted)
- May produce false positives on rare-but-normal events
- HVAC failure not currently detectable (model trained on steady-state data — v3 retraining planned)
- More sophisticated models (LSTM, Autoencoder, CUSUM) deferred to post-capstone

## 7. Risk Assessment

### Technical Risks

**Risk 1: HVAC Failure Not Detectable by Current Model**

- **Impact:** High — key demo scenario produces no alert
- **Likelihood:** Confirmed — already occurring
- **Mitigation:** Frame honestly as known limitation. Retrain model v3 with anomaly transition sequences in Week 10. Never fake detection with hardcoded threshold — this was explicitly rejected.

**Risk 2: Comparison Panel Empty at Submission**

- **Impact:** High — project fails to answer its own research question
- **Likelihood:** Medium — blocked on CentralServer implementation
- **Mitigation:** Build CentralServer as the first Week 10 task. Comparison panel is the academic core.

**Risk 3: WebSocket Performance Issues at Scale**

- **Impact:** Low — dashboard lags or drops messages with multiple nodes
- **Likelihood:** Low — three nodes tested and stable
- **Mitigation:** Chart history capped at 60 points. 60-minute stability test in Week 11.

**Risk 4: SQLite Performance for Time-Series Data**

- **Impact:** Medium — query slowdowns, storage bottlenecks
- **Likelihood:** Low
- **Mitigation:** Keep simulation duration short (<1 hour). Index timestamp columns. Batch inserts. Schema designed for future InfluxDB migration.

**Risk 5: Integration Challenges Between Components**

- **Impact:** High — components work separately but fail together
- **Likelihood:** Low
- **Mitigation:** Integration testing starting Week 10. Frequent end-to-end testing.

**Risk 6: Physics/ML Training Distribution Divergence**

- **Impact:** High — changes to physics parameters can invalidate the trained model silently
- **Likelihood:** Medium if physics parameters are modified
- **Mitigation:** Physics parameters locked. Any change to normal-operation parameters requires explicit acknowledgment and model retraining.

### Project Risks

**Risk 1: Scope creep — initial plans exceed timeline**

- **Impact:** High
- **Likelihood:** Medium
- **Mitigation:** Document clear scope in SRS. Use "Should Have" and "Could Have" for stretch features.

**Risk 2: Team member availability**

- **Impact:** Medium
- **Likelihood:** High
- **Mitigation:** Clear task assignments with deadlines. Parallel workstreams.

## 9. Appendix

### A. Glossary

- **ML:** Machine Learning — Process of training algorithms to make predictions based on learned data and pattern recognition.
- **MVP:** Minimum Viable Product — core features needed for launch
- **WebSocket:** A persistent, full-duplex communication protocol enabling real-time data streaming between client and server.
- **Isolation Forest:** An unsupervised machine learning algorithm for anomaly detection based on random partitioning of feature space.
- **Centralized Computing:** A centralized architecture where data processing occurs in one central location.
- **Edge Computing:** A distributed architecture where data processing occurs locally at or near the data source, rather than in a central server.
- **PsyEngine:** The internal name for E-Habitat's custom physics simulation engine (ThermalModel, AirflowModel, HumidityModel and their coupling inside VirtualNode).
- **seq_id:** Monotonically increasing integer assigned to each simulation step, used to synchronize edge and centralized sliding windows for valid latency comparison.
