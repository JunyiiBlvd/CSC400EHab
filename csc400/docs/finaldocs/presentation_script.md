## A NOTE ON THE PRE-RECORDED VIDEO

**Recommendation: The video should be narrated (voiceover recorded with the video).**

**Why narrated rather than live commentary:**

The live dashboard involves precise timing — anomaly injection, waiting for the 10-step ML window to fill, watching the comparison panel populate. If you narrate live while the video plays, you risk speaking too early or too late relative to what's on screen. A narrated video lets you rehearse the exact words against the exact moment each element appears, record it until it's clean, and then present with confidence knowing the demo will land. After the video ends, you pick up the script naturally — there is no awkward pause or handoff.

**Staging:** The video plays on the projected screen. You stand to the side, slightly off-center, facing the audience. You do not need to look at the screen during the video — the audience watches it, and you can look at them or slightly toward the screen. When the video ends, you step back to center.

**Video length target:** 90 seconds.

**What to record in the video:**
1. Open the dashboard (already loaded, server running, 3 nodes streaming)
2. Point out the topology hero briefly — show the animated packets
3. Show Node 1 gauges (temperature, CPU, normal readings)
4. Click "Inject Thermal Spike" on Node 1
5. Show the anomaly alert appearing in the alerts feed
6. Let the comparison panel populate — show edge latency, central latency, latency delta
7. Show the bandwidth numbers (bytes edge vs. bytes central)
8. Switch to the History tab — show the aggregate stats cards

**LOGAN:**
Good [morning/afternoon]. We're Logan, Jared, and Gavin — and this is E-Habitat.

E-Habitat is a real-time server room monitoring simulator that we built to answer a specific question about distributed systems architecture: **does edge-based anomaly detection actually outperform centralized detection?** And if so — by how much, and in what way?

*[Advance slide.]*

---

### [SLIDE 2 — The Problem]

**LOGAN:**
Here's the problem we started with.

Students in distributed systems courses learn about edge computing, latency, and bandwidth trade-offs — but they learn about them abstractly. There's no way to *see* the difference between edge and centralized architectures behaving on the same data at the same time. Existing production tools are too complex for an educational setting, and theoretical materials don't build intuition.

We wanted to fix that.

*[Advance slide.]*

---

### [SLIDE 3 — Research Question]

**LOGAN:**
So we set a clear research question — and committed to answering it with real, measured data, not estimates.

*[Read the question clearly, not rushed.]*

> "Does edge-based ML inference detect anomalies faster and with less network bandwidth than centralized polling on identical sensor data?"

We'll have an answer for you in about three minutes.

*[Advance slide.]*

---

### [SLIDE 4 — System Overview]

**JARED:**
Here's what E-Habitat does.

We simulate three virtual server room nodes — each one running coupled physics models for temperature, airflow, humidity, and CPU load. Every second, each node generates real telemetry.

That telemetry goes down two paths simultaneously. On the **edge path**, the ML model runs inside the node before anything is transmitted — the anomaly detection happens locally. On the **centralized path**, raw data is sent to a central server and inference runs there.

Both architectures run at the same time, on the same data, using the same trained model. The only thing that changes is *where* the inference happens.

*[Advance slide.]*

---

### [SLIDE 5 — Architecture Diagram]

**JARED:**
This is what that looks like structurally.

Edge sends a full frame — sensor readings plus the anomaly score and anomaly flag. Central sends only the four raw sensor values, and the inference step happens server-side after receipt.

The result: centralized transmits roughly **half the bytes** of edge — because it never sends the anomaly fields over the wire. But it adds an extra processing hop. We measure the time difference between the two detections for every anomaly injection — and that's our latency delta.

*[Advance slide.]*

---

### [SLIDE 6 — Physics Engine]

**GAVIN:**
The simulation is grounded in real physics.

We wrote a custom engine — we call it PsyEngine internally — that implements Newton's Law of Cooling for the thermal model, proportional HVAC control for airflow, and a mean-reversion humidity model coupled to temperature.

CPU load is generated by an AR(1) stochastic process — this gives it realistic volatility rather than a flat line or pure random noise.

Parameters were calibrated against three real-world datasets: the MIT CSAIL Intel Lab sensor deployment, a Kaggle HVAC dataset, and a cold source server room control dataset that Gavin sourced. The simulation doesn't behave like a math problem — it behaves like a server room.

*[Advance slide.]*

---

### [SLIDE 7 — ML Pipeline]

**GAVIN:**
The ML model is an Isolation Forest — a well-established unsupervised algorithm for anomaly detection.

We chose it for three reasons. It's fast enough to run in real time at the edge. It doesn't require labeled anomaly data to train — just normal operation. And its output is an interpretable score, not a black-box binary.

Feature extraction uses a 10-step sliding window. For each of the four sensor signals — temperature, airflow, humidity, and CPU load — we compute mean, variance, and rate of change. That gives us 12 features per inference call.

The model was trained on a hybrid dataset of 22,239 rows: 45% synthetic physics data, 22% MIT CSAIL, 17% Kaggle HVAC, and 16% cold source server room data. The cold source dataset had no humidity column — we initially filled it with arbitrary parameters, which the professor flagged. We audited the data, recalibrated humidity to match MIT CSAIL statistics, and retrained. The new model improved recall on real anomaly windows from 95.96% to 100%.

The detection threshold is 0.15. During normal operation, the lowest score we've measured is about 0.23 — so there's an eight-point safety margin. Zero false positives in baseline testing.

*[Advance slide.]*

---

### [SLIDE 8 — Real-World Validation]

**GAVIN:**
One question that came up early: does this model actually work on real anomalies? Or does it only work on the synthetic data it was tuned against?

We ran a full evaluation: 1,732 MIT CSAIL anomaly windows as positives, and 134,516 in-distribution normal windows as negatives.

**Recall: 100%.** Every anomaly detected. **Precision: 97.58%** — only 43 normal windows misclassified out of 134,516. **F1: 98.77%. AUC-ROC: 1.0.**

The score distributions barely overlap: anomaly windows average +0.005, normal windows average +0.269, with our threshold at 0.15 sitting cleanly between the two populations.

One finding worth mentioning: when we scored the cold source dataset's normal windows against the model, it flagged 94.9% of them. That sounds alarming, and it reveals a real data quality issue. The cold source dataset has no humidity column — we imputed it by sampling i.i.d. from N(38.63, 7.21), matching MIT CSAIL's long-run marginal distribution. But real sensors don't jump 7 units of humidity between consecutive readings — they drift slowly. When you sample i.i.d. from σ=7.21, the expected sliding-window variance is σ² ≈ 52. The observed cold source humidity variance is 49. The math matches: we generated white noise that looked like humidity values, and the model correctly rejected it. The lesson: matching a marginal distribution is not the same as matching a time-series. We excluded cold source from the FP metrics and documented it.

*[Advance slide.]*

---

### [SLIDE 9 — Injection Scenarios]

**JARED:**
We support three anomaly injection scenarios.

A **thermal spike** overrides CPU load to 100% for two minutes. Temperature rises, airflow struggles to compensate, and the ML model detects it through the thermal and CPU feature channels.

An **HVAC failure** ramps airflow down to zero over 15 steps. We learned early that an instantaneous cutoff wasn't detectable — the 10-step sliding window didn't fill with enough deviant readings to exceed threshold. The ramped design gives the window time to build a strong signal.

A **coolant leak** ramps humidity up by 2.5% per step for 20 steps, capping at 85%. The humidity variance and rate-of-change features diverge, and the model flags it.

All three are injectable from the dashboard with a single button click.

*[Advance slide.]*

**LOGAN:**

### [SLIDE 12 — Results: Latency]

**LOGAN:**
Here are the latency numbers.

Both architectures detect anomalies in the two-and-a-half to four-second range after injection. That's the window fill time — it takes 10 seconds of data for the sliding window to build a meaningful feature vector, but if injection happens mid-window, detection is faster.

Edge consistently detects **38 to 42 milliseconds earlier** than centralized. The delta varies between 10 and 1,000 milliseconds depending on the window fill state at the moment of injection — that variance is correct and academically meaningful. The direction is always the same: edge first.

In absolute terms, 38 milliseconds is small. In low-latency applications — power grid anomalies, medical telemetry, autonomous systems — it matters. For this project, the important finding is the direction and the mechanism: edge is faster because it skips the transmission hop.

*[Advance slide.]*

---

### [SLIDE 13 — Results: Bandwidth]

**JARED:**
The bandwidth result is more significant in magnitude.

Centralized transmits roughly **half the bytes** of edge — because the central path sends only the four raw sensor fields. The edge path sends those four fields plus the anomaly score and the anomaly flag.

At three nodes running continuously, centralized generates about 98 kilobytes of data versus 204 kilobytes for edge — a **52% reduction**.

At scale — thousands of IoT nodes, continuous streaming — that difference compounds fast. A centralized architecture can reduce network load by more than half without sacrificing much detection speed.

*[Advance slide.]*

---

### [SLIDE 14 — Known Limitations]

**GAVIN:**
We want to be clear about what this system doesn't do.

HVAC failure required a ramped injection design — an instantaneous fan cutoff wasn't reliably detectable with the current 10-step window. The "HVAC failure" scenario in the demo is a slow degradation, not a sudden failure. That's an honest engineering constraint.

The model detects deviations from normal — it doesn't diagnose root causes. "Something is wrong" is the answer, not "the fan failed" or "there's a coolant leak."

And the latency delta is small. 38 to 42 milliseconds may not be meaningful for every application. We're not claiming edge is dramatically faster — we're showing it's consistently faster, and explaining why.

*[Advance slide.]*

---

### [SLIDE 15 — Lessons Learned]

**LOGAN:**
A few things we'd do differently.

The cold source dataset — 16% of our training data — had no humidity column. We filled it with arbitrary parameters initially. The professor flagged it. When we tried to fix it by using real MIT CSAIL humidity statistics, recall collapsed from 95.96% to 15.53% in two successive retrains. It looked like the calibration itself was broken. The real problem was that our validation script was using the model's internal threshold, not the 0.15 threshold we actually deploy at runtime. Two different numbers that look like they should be the same — they're not. Fixing the validation gate and retraining pushed recall to 100%. That's now documented so it can't happen again.

The bigger lesson: physics and model are tightly coupled. You can't change normal operation parameters without retraining. We learned to treat the model as a dependency, not an afterthought.

We also learned to build the comparison panel first. It's the academic core — the thing that answers the research question. We committed to it early in Phase 3. If we'd left it for last, we might have run out of time.

*[Advance slide.]*

---

### [SLIDE 16 — Future Work]

**JARED:**
Where this goes next:

A retrained model — version 3 — would include anomaly sequences in training data, enabling confidence scores and root-cause hints rather than just binary detection.

Multi-user support would let an entire classroom inject scenarios on different nodes simultaneously — making it a true lab environment, not just a demo tool.

And migrating time-series storage to InfluxDB would remove the SQLite write-rate ceiling, enabling longer sessions and production-scale data volumes.

*[Advance slide.]*

### [SLIDE 17 — Conclusion]

**LOGAN:**
To close — E-Habitat answers its research question.

Edge-based ML inference detects anomalies 38 to 42 milliseconds faster than centralized polling. Centralized transmits 52% less bandwidth.

Both architectures run on the same data, the same physics, the same model. The comparison is fair. The numbers are real — every latency and byte count you saw came from live simulation measurements.

The Isolation Forest model achieves **100% recall, 97.58% precision, 98.77% F1, and a 1.0 AUC-ROC** on held-out real-world sensor data — trained on four data sources, calibrated against real sensor distributions, and validated with metrics that hold up.

If you're weighing edge versus centralized for an anomaly detection deployment — the answer depends on your application. But now you can see exactly where the trade-off lives, and measure it.

Thank you.
